# Customer Churn Prediction & Retention Analysis System

A machine learning system that predicts whether a customer is likely to churn and identifies the key
factors driving that churn — enabling proactive, targeted retention action instead of guesswork.

This is an intermediate-level, business-aware ML project: the focus isn't just "can we predict churn"
but "can we explain *why* customers churn well enough for a business to act on it."

## Project Overview

| | |
|---|---|
| **Problem type** | Binary classification |
| **Target variable** | `Churn` (Yes / No) |
| **Dataset** | [Telco Customer Churn](https://github.com/IBM/telco-customer-churn-on-icp4d) — 7,043 customers, 21 features |
| **Primary metric** | Recall (catching at-risk customers matters more than raw accuracy) |
| **Best model** | Gradient Boosting — 85% recall on churners, 0.50 precision, 73% accuracy |

## Workflow

The notebook follows a 10-step business-to-deployment pipeline:

1. **Business Understanding** — defines churn, its revenue impact, and why Recall/F1 are prioritized over raw accuracy
2. **Data Cleaning** — fixes the `TotalCharges` string-to-numeric conversion, imputes the 11 true zero-tenure cases, checks for duplicates/invalid entries
3. **Feature Engineering** — adds `AvgRevenuePerMonth`, `TenureGroup`, `ServiceCount`, premium/streaming usage flags, and a `ContractRiskScore`
4. **EDA** — churn rate by contract type, tenure, monthly charges, correlation heatmap, churn by individual service
5. **Imbalance Handling** — SMOTE oversampling + class-weighted models (churn is ~26.5% of the dataset)
6. **Model Building** — Logistic Regression, Random Forest, Gradient Boosting (XGBoost), and SVM, each in a leak-safe preprocessing pipeline
7. **Evaluation** — Accuracy, Precision, Recall, F1, ROC-AUC, and confusion matrix, compared across all four models
8. **Interpretation** — feature importance (tree-based models) and SHAP values to explain *why* customers churn
9. **Deployment** — a Streamlit app stub (`app.py`) with a customer input form, churn probability, risk category, and key drivers
10. **Business Recommendations** — concrete retention strategies, model limitations, and next steps

## Key Findings

- **Contract type is the strongest lever** — month-to-month customers churn far more than annual/biennial customers.
- **Risk is front-loaded** — churn is concentrated in a customer's first 12 months.
- **Price sensitivity** — higher monthly charges correlate with higher churn.
- **Support add-ons reduce churn** — customers without OnlineSecurity/TechSupport are more likely to leave.
- **Fiber-optic internet customers churn more than DSL customers**, which is worth a follow-up business conversation on pricing or service quality.

## Model Limitations

- Trained on a single static snapshot — doesn't capture behavioral *trends* (e.g., usage decline, recent complaints) that would likely improve predictions.
- SMOTE-generated synthetic samples improve recall but don't add genuinely new information; performance is always validated on real, untouched holdout data.
- Built on telecom-style data — pricing tiers and contract structures differ across industries, so the model would need retraining (not just reuse) elsewhere.
- Uses the default 0.5 classification threshold; a production deployment should tune this against the actual cost of a missed churner vs. an unnecessary retention offer.

## Future Improvements

- Add behavioral/usage-trend features (e.g., usage change over the last 3 months, support ticket recency)
- Tune the decision threshold against a real business cost matrix
- Try stacking/ensembling the top-performing models
- Add periodic retraining and drift monitoring once deployed
